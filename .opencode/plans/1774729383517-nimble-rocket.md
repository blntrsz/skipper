# Docker sandbox plan

## Recommended approach

- Keep the existing workspace use cases unchanged and make sandbox selection a runtime composition concern.
- Add a new `DockerSandboxService` that implements `SandboxService` in parallel with the current worktree adapter.
- Keep `worktree` as the default sandbox, and add a global CLI flag `--sandbox <worktree|docker>` with default `worktree`.
- Reuse the current `FileSystemService` and host workspace paths; in Docker mode, mount the resolved host workspace into a fixed container path and translate host `cwd` values inside the Docker adapter.
- Refactor runtime assembly so the CLI selects which `SandboxService` layer to provide via DI before command execution starts.

## Interfaces

- Add a sandbox selection type, e.g. `type SandboxKind = "worktree" | "docker"`.
- Add a global CLI flag definition for sandbox selection with default `worktree`.
- Keep `SandboxService` unchanged unless implementation reveals a hard requirement for Docker-specific error reasons.
- Add Docker adapter-local config/constants for image name, container mount path, shell command, and container naming.

## Critical files

- Modify `packages/cli/src/common/global-flags.ts`
- Modify `packages/cli/src/command.ts`
- Modify `packages/cli/src/index.ts`
- Refactor `packages/core/src/runtime/local-work-tree.runtime.ts` into a runtime/layer selector or factory
- Add `packages/core/src/workspace/adapter/docker-sandbox.service.ts`
- Reuse `packages/core/src/workspace/port/sandbox.service.ts`
- Reuse existing workspace use cases:
  - `packages/core/src/workspace/use-case/init-workspace.use-case.ts`
  - `packages/core/src/workspace/use-case/attach-workspace.use-case.ts`
  - `packages/core/src/workspace/use-case/run-command-in-workspace.use-case.ts`
  - `packages/core/src/workspace/use-case/destroy-workspace.service.ts`

## High-level implementation steps

1. Add the global sandbox flag and default it to `worktree`.
2. Introduce a shared sandbox selection type that the CLI/runtime can use.
3. Extract runtime composition so shared layers remain common while `SandboxService` is selected dynamically.
4. Implement `DockerSandboxService` against the existing `SandboxService` interface.
5. In the Docker adapter, manage container lifecycle for `init`/`destroy`, mount the resolved host workspace, and route `execute`/`attach` into the container using translated working directories.
6. Keep worktree behavior unchanged when the flag is omitted.
7. Add tests for runtime selection, default behavior, and Docker-specific sandbox flows.

## Implementation notes

- First pass should keep Docker scoped to sandbox execution and attachment, not introduce a Docker-specific filesystem abstraction.
- Container naming should derive from `ProjectModel` so branch workspaces remain isolated and cleanup stays idempotent.
- `attach`/`detach` semantics should be defined during implementation, but should still satisfy the current `SandboxService` contract.
- The Docker image/tooling set should be explicit so commands run with the expected shell and developer tools.

## Verification

- Confirm CLI behavior is unchanged with no flag and still uses the worktree sandbox.
- Confirm `--sandbox docker` selects the Docker implementation through DI.
- Verify workspace init still uses the same host paths from `FileSystemService` for main and branch workspaces.
- Verify command execution runs inside Docker against the mounted workspace and correct working directory.
- Verify attach/destroy work cleanly and repeated cleanup is safe.
- Run targeted tests for CLI/runtime selection and sandbox behavior.
- Run `bun run check:fix` from the repository root.
