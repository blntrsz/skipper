Root cause
- `sk run` reaches `src/Agent/Service.ts`, then calls `RepositoryPath.make({ repository, branch: "main" })`.
- `src/domain/RepositoryPath.ts` expects a repo-name string and does `join(REPOSITORY_ROOT, repository)`.
- That sends an object into `path.join`, causing `TypeError: The "paths[1]" property must be of type string, got object`.

Recommended approach
1. Fix the bad call in `src/Agent/Service.ts` to pass only the repository name string into `RepositoryPath.make(...)`.
2. Re-check that this flow should use the repository root path, not a branch worktree path; keep `WorkTreePath.make(...)` only for branch-specific sandbox flows.
3. Harden `src/domain/RepositoryPath.ts` so misuse is easier to catch:
   - keep the input strictly string-typed
   - optionally add a small runtime guard with a clear error message if a non-string slips through
4. Audit all `RepositoryPath.make(` call sites and normalize any other object-vs-string misuse.
5. If needed, add short JSDoc to `src/domain/RepositoryPath.ts` and `src/domain/WorkTreePath.ts` clarifying repo-root vs worktree responsibilities.

Critical files
- `src/Agent/Service.ts`
- `src/domain/RepositoryPath.ts`
- maybe `src/domain/WorkTreePath.ts` if docs/type contrast helps

Verification
- Run `bun test`.
- Run a focused CLI repro: `bun run ./src/Cli.ts run "prompt" --repository <repo>` and confirm the `paths[1]` error is gone.
- Run `sk run "prompt" --repository <repo>` against the built CLI.
- Run `sk run "prompt"` without `--repository` and confirm fuzzy repo selection still works.
- Grep for `RepositoryPath.make(` and confirm all callers pass a string.

Risks
- Tightening types may expose other bad call sites.
- If this code actually intended a worktree path, fixing only the crash would still leave wrong cwd semantics; verify expected cwd while implementing.
