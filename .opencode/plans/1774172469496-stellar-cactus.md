Fix CLI DI typing in `packages/cli/src/index.ts`.

Approach

- Update `packages/cli/src/index.ts` to follow Effect CLI runtime composition used in local Effect refs.
- Keep command tree as-is: `Command.make("skipper")`, `Command.withSubcommands([...])`, `Command.run({ version })`.
- Replace raw `localWorkTreeLayer` pipe stage with `Effect.provide(localWorkTreeLayer)`.
- Add/import `Effect` from `effect` in the same file.
- Keep `BunRuntime.runMain` last.

Why

- `Command.run(...)` returns an `Effect` with remaining env requirements.
- `localWorkTreeLayer` is a `Layer`, not a unary pipe fn.
- Passing the raw layer into `pipe(...)` breaks inference and surfaces the misleading `=> never` error.
- Local Effect refs use `Command.run(...).pipe(Effect.provide(layer), Runtime.runMain)`.

Critical files

- `packages/cli/src/index.ts`
- Ref only: `packages/core/src/runtime/local-work-tree.runtime.ts`
- Ref only: `.agents/effect-smol/packages/tools/bundle/src/bin.ts`

Verification

- Run repo checks: `bun run check:fix`
- Run CLI package typecheck: `bun run typecheck` in `packages/cli`
- Smoke test entrypoint/help path: `bun run ./packages/cli/src/index.ts --help`
- If needed, build CLI binary: `bun run build` in `packages/cli`

Risks

- If TS still reports missing env, inspect the remaining `R` after `Effect.provide(localWorkTreeLayer)`; that would mean `localWorkTreeLayer` misses part of `Command.Environment` or app services.
- Avoid changing subcommand defs unless new type errors prove a second issue.
